## Задача 2 (*)

Задеплоен python-форк из ДЗ 5 (`shvirtd-example-python`) в Swarm-кластер.

### Адаптация проекта под Swarm

Обычный `docker compose` (использованный в ДЗ 5) не переносится в Swarm один в один — потребовались изменения:

| В ДЗ 5 (compose) | В ДЗ 6 (Swarm) | Почему |
|---|---|---|
| `build:` контекст | готовый образ из Docker Hub | Swarm не умеет собирать образы — только тянуть готовые |
| bind mount конфигов haproxy/nginx | **Docker Configs** | bind mount работает только если файл есть на конкретной ноде назначения; конфиг как объект Swarm доступен любой ноде |
| статичный IP `172.20.0.5` в haproxy.cfg | DNS-имя сервиса `web` | overlay-сеть не поддерживает фиксированные IP так же, как bridge; используется встроенный Docker DNS |
| `network_mode: host` у nginx | обычная публикация порта (`ports: 8090:8090`, `mode: ingress`) | Swarm Routing Mesh сам раскидывает трафик на нужную ноду |
| `version: '3'` bridge-сеть `backend` | `driver: overlay` | overlay — единственный тип сети, работающий между разными физическими нодами |

### Образ приложения

```bash
docker tag shvirtd-example-python-web zolotkoff/shvirtd-python:swarm
docker push zolotkoff/shvirtd-python:swarm
```

### Docker Configs

```bash
docker config create haproxy_cfg haproxy.cfg
docker config create nginx_default_conf nginx-default.conf
docker config create nginx_conf nginx.conf
docker config create mysql_init mysql-init-auth.sh
```

### stack.yml

```yaml
version: "3.8"

services:
  web:
    image: zolotkoff/shvirtd-python:swarm
    networks:
      - backend
    environment:
      DB_HOST: db
      DB_USER: app
      DB_PASSWORD: QwErTy1234
      DB_NAME: virtd
    deploy:
      replicas: 1
      restart_policy:
        condition: on-failure

  db:
    image: mysql:8.0
    networks:
      - backend
    environment:
      MYSQL_ROOT_PASSWORD: YtReWq4321
      MYSQL_DATABASE: virtd
      MYSQL_USER: app
      MYSQL_PASSWORD: QwErTy1234
    configs:
      - source: mysql_init
        target: /docker-entrypoint-initdb.d/01-auth.sh
        mode: 0755
    deploy:
      replicas: 1
      restart_policy:
        condition: on-failure

  reverse-proxy:
    image: haproxy:2.4
    networks:
      - backend
    configs:
      - source: haproxy_cfg
        target: /usr/local/etc/haproxy/haproxy.cfg
    deploy:
      replicas: 1
      restart_policy:
        condition: on-failure

  ingress-proxy:
    image: nginx:latest
    networks:
      - backend
    ports:
      - target: 8090
        published: 8090
        mode: ingress
    configs:
      - source: nginx_default_conf
        target: /etc/nginx/conf.d/default.conf
      - source: nginx_conf
        target: /etc/nginx/nginx.conf
    deploy:
      replicas: 1
      restart_policy:
        condition: on-failure

networks:
  backend:
    driver: overlay

configs:
  haproxy_cfg:
    external: true
  mysql_init:
    external: true
  nginx_default_conf:
    external: true
  nginx_conf:
    external: true
```

### Решённая проблема: 503 от HAProxy

После первого деплоя `curl http://127.0.0.1:8090` возвращал `503 Service Unavailable`. Причина — в исходном `haproxy.cfg` стояла директива `default-server init-addr none`, означающая «не резолвить адрес сервера вообще» (в ДЗ 5 она была безвредна, так как там использовался не хостнейм, а готовый статичный IP). После замены IP на DNS-имя `web` директива блокировала резолвинг.

**Решение** — добавлена секция `resolvers`, указывающая на встроенный Docker DNS (`127.0.0.11:53`), и `server web web:5000 check resolvers docker init-addr none` для динамического резолвинга:

```
resolvers docker
  nameserver dns1 127.0.0.11:53
  resolve_retries 3
  timeout resolve 1s
  timeout retry   1s
  hold valid      10s

backend http_back
  server web web:5000 check resolvers docker init-addr none
```

Конфиг обновлён без пересоздания стека:
```bash
docker config create haproxy_cfg_v2 haproxy.cfg
docker service update --config-rm haproxy_cfg --config-add source=haproxy_cfg_v2,target=/usr/local/etc/haproxy/haproxy.cfg shvirtd_reverse-proxy
```

### Проверка Routing Mesh

Запрос на **любую** ноду кластера (включая worker-ноды, где физически nginx может и не находиться) корректно маршрутизируется на контейнер `ingress-proxy`:

```bash
curl -L http://127.0.0.1:8090        # с мастера
curl -L http://93.77.181.237:8090    # через worker2
curl -L http://111.88.250.45:8090    # через worker1
```

Все три запроса вернули корректный ответ `"TIME: ..., IP: ..."` — подтверждение, что Swarm Routing Mesh сам нашёл контейнер `nginx`, независимо от того, на какую ноду пришёл запрос.

### Проверка БД

`db`-реплика оказалась на ноде `swarm-worker2` — `docker ps` локально её не покажет, поэтому подключение выполнено через `docker stack ps` (поиск ноды) + SSH на неё:

```bash
docker stack ps shvirtd --filter "name=shvirtd_db" --no-trunc
ssh zolotkoff@93.77.181.237
CONTAINER_ID=$(sudo docker ps -q -f name=shvirtd_db)
sudo docker exec -ti $CONTAINER_ID mysql -uroot -pYtReWq4321 -e "USE virtd; SELECT * FROM requests LIMIT 10;"
```
